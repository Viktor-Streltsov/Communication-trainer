## Общее описание продукта (контекст для Cursor)

Мобильное приложение — тренажёр общения. Пользователь выбирает роль/сценарий
(собеседование, разговор с руководителем, партнёром, родителем, другом) и уровень
строгости, после чего ведёт диалог (текст + голос) с ИИ-персонажем. После сессии
получает разбор: что было сказано нечётко, где могли неправильно понять, как
переформулировать точнее.

Стек: React + Capacitor (мобильное приложение) + FastAPI (Python, бэкенд) + PostgreSQL +
LLM API для диалогов и разбора.

## Этап 1 — Бэкенд-прототип (FastAPI + LLM)

```
Работаем в папке backend/. Создай прототип FastAPI-бэкенда для тренажёра общения.

Требования:
1. Структура:
   backend/
   ├── app/
   │   ├── main.py
   │   ├── routers/
   │   │   └── chat.py
   │   ├── services/
   │   │   ├── llm_client.py
   │   │   └── persona_prompts.py
   │   └── config.py
   ├── requirements.txt
   └── .env.example

2. config.py — читает переменные окружения через pydantic-settings. Не хардкодь
   ключи, используй .env (добавь .env в .gitignore). Переменные:
   - llm_provider: str = "groq" (по умолчанию Groq — бесплатный API, полностью
     совместимый с форматом OpenAI, так что клиент можно писать один раз)
   - llm_api_key: str
   - llm_model: str = "llama-3.3-70b-versatile"
   - llm_api_base_url: str = "https://api.groq.com/openai/v1"
   В .env.example пропиши эти же значения как заготовку (кроме ключа — вместо
   него плейсхолдер) с комментарием, что для смены провайдера (OpenAI/OpenRouter/
   другой) достаточно поменять llm_provider/llm_api_base_url/llm_model, так как
   Groq использует OpenAI-совместимый формат запросов.

3. persona_prompts.py — словарь с 3 готовыми сценариями-персонажами:
   - "interview_strict" (жёсткий HR на собеседовании)
   - "boss_business" (строгий руководитель, деловой разговор)
   - "family_parent" (разговор с заботливым, но настойчивым родителем)
   Для каждого — системный промпт: кто персонаж, как себя ведёт, как реагирует
   на нечёткие/двусмысленные фразы пользователя (переспрашивает, показывает
   непонимание — в этом и есть тренировочный эффект).

4. llm_client.py — обёртка над вызовом LLM API через httpx, используя формат
   OpenAI Chat Completions API (POST {llm_api_base_url}/chat/completions,
   заголовок Authorization: Bearer {llm_api_key}, тело с messages/model) —
   Groq и большинство других провайдеров понимают именно этот формат, поэтому
   клиент не нужно переписывать под конкретного провайдера.

5. routers/chat.py — эндпоинты:
   - POST /chat/start — принимает scenario_id, создаёт сессию, возвращает session_id
     и первую реплику персонажа
   - POST /chat/message — принимает session_id и текст пользователя, возвращает ответ
     персонажа
   Хранить историю диалога пока что можно в памяти (простой dict по session_id),
   базу данных подключим на следующем этапе.

6. main.py — подключает роутер, CORS для локальной разработки мобильного приложения.

7. Добавь requirements.txt (fastapi, uvicorn, pydantic-settings, httpx, python-dotenv).

После создания покажи, как локально запустить сервер (uvicorn) и протестировать
эндпоинты через curl или httpie — приведи готовые примеры команд. Отдельно укажи,
что ключ для .env нужно взять на console.groq.com (API Keys → Create API Key,
без привязки карты).
```

## Этап 3 — Разбор сессии (AI-коуч)

```
Работаем в backend/. Добавь функциональность разбора завершённой сессии.

4. Добавь services/analysis.py с промптом для коуча (вынеси отдельно от
   persona_prompts.py — это другая роль LLM, не персонаж диалога).

Покажи пример полного цикла через curl: start → message (несколько раз) → end → review.
```

## Этап 3.4 — Пять сценариев + итог разговора с оценкой

Две отдельные, но связанные доработки: расширяем количество персонажей с 3 до 5,
и делаем разбор в конце сессии более полезным — не только "какие фразы были
неудачными", но и как это звучало в целом (неуверенно, нервно, извиняющимся
тоном и т.п.), плюс общий % успеха сессии.

**Новые персонажи (добавляются к interview_strict, boss_business, family_parent):**
- `partner_conversation` — партнёр/супруг(а): разговор о важном для отношений,
  персонаж не принимает уклончивое "всё нормально", хочет искренности про чувства
- `friend_casual` — дружеская беседа: низкая строгость, но тоже проверяет, умеет
  ли пользователь ясно донести мысль (например, попросить о помощи или поделиться
  новостью), не в лоб, а по-дружески переспрашивая, если что-то неясно

```
Работаем в backend/. Добавь два новых сценария-персонажа в
app/services/persona_prompts.py, в том же стиле и структуре, что уже есть у
interview_strict, boss_business, family_parent (Persona dataclass: scenario_id,
display_name, system_prompt, opening_line):

1. "partner_conversation" — заботливый, но эмоционально включённый партнёр/супруг.
   Разговор о чём-то важном для отношений (не обязательно конфликт — может быть
   про планы, чувства, недопонимание). Персонаж не удовлетворяется уклончивыми
   ответами вроде "всё нормально" или "как обычно" — переспрашивает мягко, но
   настойчиво, показывает, что не понимает, если пользователь говорит расплывчато
   про свои чувства или намерения.

2. "friend_casual" — обычный близкий друг, непринуждённый тон, минимум давления.
   Сценарий: пользователь хочет о чём-то попросить друга или поделиться новостью.
   Друг задаёт уточняющие вопросы по-доброму, если пользователь путается или
   не может ясно сформулировать, чего хочет — не критикует, а просто искренне
   не понимает и просит объяснить иначе.

Также обнови backend/seed.py: добавь оба новых scenario_id в словарь
SCENARIO_META с role_type ("partner" и "friend" соответственно) и strictness
("medium" для partner_conversation, "low" для friend_casual). После обновления
запусти python seed.py заново — существующие записи не дублируются
(seed.py уже это проверяет), новые два добавятся.
```

**Итог разговора: эмоциональный/поведенческий разбор + % успеха**

Сейчас `analyse_session` оценивает только формулировки (сильные стороны, слабые
фразы, как перефразировать). Добавляем: (1) общее наблюдение о манере речи —
неуверенность, извиняющийся тон, потеря нити разговора, излишняя многословность
и т.п., не привязанное к конкретной цитате, и (2) числовую оценку успеха сессии.

```
Работаем в backend/. Расширь разбор сессии в app/services/analysis.py и
app/models/session_review.py следующим образом:

1. В models/session_review.py добавь колонку:
   success_score: Mapped[int] = mapped_column(nullable=False, default=0)
   (значение 0-100)
   Создай новую Alembic-миграцию (alembic revision --autogenerate -m "add
   success_score to session_reviews") и примени её (alembic upgrade head).

2. В services/analysis.py обнови COACH_SYSTEM_PROMPT: добавь в JSON-схему ответа
   три новых поля:
   - "overall_impression": строка — общее наблюдение о манере речи пользователя
     за всю сессию (например: "Ты часто извинялся и использовал неуверенные
     формулировки вроде «наверное» и «может быть» — это звучит как нервозность,
     даже если по сути ты был прав"). Пиши обращение на "ты", по-человечески,
     а не канцелярским языком отчёта.
   - "success_score": число от 0 до 100 — насколько успешно, по мнению коуча,
     прошёл разговор с точки зрения того, был ли пользователь понят правильно
     и donёс ли то, что хотел. Учитывай ясность формулировок, уверенность тона
     (по тексту — повторы, извинения, обрывы мысли), и то, добился ли пользователь

   - "motivational_message": строка — короткая (1-2 предложения) тёплая фраза

     произошло в диалоге (например: "Ты справился с самым сложным вопросом про
     причину увольнения — именно там многие теряются. В следующий раз попробуй
     так же чётко ответить и на вопрос про зарплату"). Даже при низком

3. Обнови AnalysisResult (добавь overall_impression: str, success_score: int,
   motivational_message: str) и функцию analyse_session — распарси все три новых
   поля из ответа LLM, с разумным fallback (success_score=50,
   overall_impression="" и motivational_message="Ты прошёл через непростой
   разговор — это уже само по себе тренировка" если LLM не вернул поле).

4. В routers/chat.py обнови ReviewResponse (добавь overall_impression: str,
   success_score: int, motivational_message: str) и оба места, где формируется
   ReviewResponse (/chat/end и GET /chat/session/{id}/review) — прокинь новые
   поля из SessionReview/AnalysisResult. Не забудь добавить колонку

Не меняй существующую логику strengths/weak_points/suggested_phrasings —
```

## Этап 3.5 — Дизайн-система «Тихий кабинет» (перед версткой UI)

Концепция: терапевтическая сдержанность вместо гейміфикации. Спокойный кабинет,
где никто не оценивает — только пользователь сам видит свой прогресс. Никаких
процентов и ярких бейджей: строгость сценария называется словом, а не цифрой,
интерфейс дышит свободным пространством.

**Токены:**
- Фон приложения (основной): `#EAE7DF` (бледный камень)
- Поверхность карточек: `#F5F3EC` (тёплый бумажный белый)
- Основной текст: `#2B332C` (глубокий тёмно-зелёный, не чёрный)
- Вторичный текст: `#6E6A5E` (приглушённый тёплый серый)
- Акцент «деловые/строгие сценарии»: `#7C9473` (приглушённый шалфей)
- Акцент «близкие люди/тёплые сценарии»: `#9D7B72` (пыльная глина)
- Кнопка основного действия: фон `#3C4A3D` (глубокий лесной), текст `#F0EEE5`
- Заголовки/дисплей: serif с мягким характером — Newsreader (курсив 500 для
  акцентных фраз, прямой 600 для заголовков)
- Основной текст: Work Sans (400/500)
- Радиус углов карточек: 10px, без теней и градиентов — плоские поверхности
- Между блоками — заметно больше воздуха, чем в обычных приложениях (не экономь
  на отступах, это часть ощущения «не спешить»)

**Сигнатурный элемент:** «дышащая точка» — мягко пульсирующий кружок (плавное
увеличение/уменьшение через CSS-анимацию, 3-4 секунды на цикл), который
показывается перед началом любой сессии как короткий ритуал «выдохни». Уровень
строгости сценария везде обозначается словом (мягкая / средняя / строгая) рядом
с небольшим закрашенным кружком акцентного цвета — без процентов и чисел.

```
   - accent-warm: '#9D7B72' (близкие люди/тёплые сценарии)

2. Подключи шрифты через Google Fonts: Newsreader (500 italic, 600) для

   - Название сценария — font-display, подзаголовок — font-sans text-secondary

   персонажа вместо цветной пилюли-процента.

Не добавляй ничего сверх описанного — ни процентов, ни ярких цветов, ни теней,
ни лишней анимации кроме BreathingDot.
```

## Этап 3.6 — Пролог (онбординг) и домашний экран

Идея: первое впечатление — как короткий терапевтический ритуал, а не форма
регистрации и не гейміфицированный дашборд со статистикой. Домашний экран
тоже остаётся тихим: без крупных чисел и счётчиков, только мягкое напоминание
о том, где пользователь остановился.

**Экраны пролога (показываются один раз, при первом запуске):**
1. «Прежде чем начать» — курсивный заголовок font-display, спокойная фраза
   ("выдохни, никто не оценивает — только ты"), BreathingDot по центру,

## Примечания по ходу работы

- После каждого этапа — открой приложение/запусти сервер и убедись, что реально
  работает, прежде чем копировать следующий промпт.
- Если Cursor на каком-то этапе делает не то — не пытайся чинить всё сразу большим
  промптом, лучше уточни точечно: "не трогай X, только исправь Y".
- Ключи LLM API и другие секреты — всегда через .env, никогда не проси Cursor
  "просто вписать ключ в код для теста", даже временно.
